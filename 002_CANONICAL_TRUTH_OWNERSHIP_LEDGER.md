# CANONICAL TRUTH OWNERSHIP LEDGER

## 1. purpose
This ledger fixes which root domain should be treated as the canonical owner
for each major truth category.

## 2. truth ownership map

### whole-system premise / worldview / foundational doctrine
- canonical owner: 00.foundation

### civilization-wide common operating meaning
- canonical owner: 01.civilization-os

### persona identity / expression / secretary presentation meaning
- canonical owner: 02.persona-os

### business app operation / share / sync / send / business-side execution boundary
- canonical owner: 03.business-os

### real-world life records / planning / life support application meaning
- canonical owner: 04.life-os

### game runtime / game-domain rules / game experience structures
- canonical owner: 05.game-os

### streaming platform / creator-channel-content distribution / donation-social boundaries
- canonical owner: 06.streaming-os

### app-specific product flows / app-specific UX / app-local feature packaging
- canonical owner: 07.applications

### official portal / official public entry / official navigation / discoverability
- canonical owner: 08.civilization-portal-site

### static visual works / publishing works / asset rights / static-work marketplace / library-reader-viewer continuity / exhibition eligibility source
- canonical owner: 10.staticart-os

## 3. conflict rule
If two domains appear to define the same thing, prefer:

1. foundation
2. canonical OS owner
3. application-local specialization

Applications may specialize,
but they should not overwrite upper canonical truth without explicit governance.

## 4. root usage rule
- use OS folders for canonical cross-app meanings
- use applications for concrete product realizations
- use portal for official entry/navigation/public route meanings
- use StaticArtOS for static-work asset truth, rights, entitlement reflection, library/reader/viewer continuity, and Exhibition Builder eligibility source
