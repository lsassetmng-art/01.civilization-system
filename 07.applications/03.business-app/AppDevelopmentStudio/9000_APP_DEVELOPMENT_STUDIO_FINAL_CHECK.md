# ============================================================
# 1. DEFINITION
# ============================================================

App Development Studio is an independent application intended primarily
for business-oriented application design, generation, review,
approved apply, Git-linked delivery, testing support,
bug/version tracking, multilingual operation,
and internal project progress management.

It is not an internal BusinessOS domain.
It is a standalone product that may integrate with BusinessOS,
Civilization-side distribution routes, Git repositories,
local workspaces, storage, and optional remote execution routes.

# ============================================================
# 2. PURPOSE
# ============================================================

The purpose of App Development Studio is to let a user create,
refine, control, version, test, and deliver one business-oriented application
inside a single product, with configurable automation.

The application is designed to support:
- design authoring
- imported design analysis
- full-generation input processing
- Java code generation
- SQL generation
- reusable component reuse
- connection-profile-aware DB targeting
- review / approval / apply control
- Git commit / push / tag workflows
- Java unit test auto generation
- integration-and-beyond test item sheet generation
- bug management
- artifact versioning
- multilingual UI and outputs
- test progress management
- project progress management
- optional build execution

# ============================================================
# 3. POSITION
# ============================================================

App Development Studio is positioned as:

- independent application
- business-oriented development tool
- Civilization-sellable product
- BusinessOS-integratable product
- Git-integrated engineering application
- configurable rule-driven generation application

It is not:
- BusinessOS itself
- a fixed-rule generator only
- a source-of-truth owner for all target systems
- a mandatory build executor
- a mandatory manual-review-only product

# ============================================================
# 4. INPUT MODES
# ============================================================

The user may choose how to create or provide design input.

Supported design input modes:
- free_input
- guided_input
- standard_template
- full_generation_format
- imported_design_documents

Meaning:
- free_input:
  user writes freely
- guided_input:
  system guides section-by-section entry
- standard_template:
  user fills a standard structured design template
- full_generation_format:
  user fills a generation-ready format intended for full generation
- imported_design_documents:
  user imports existing design documents for parsing and assessment

# ============================================================
# 5. GENERATION READINESS DECISION
# ============================================================

When the user imports or creates design content,
the application performs internal readiness assessment.

Assessment dimensions include:
- completeness
- granularity
- ambiguity
- dependency resolution
- reusable component readiness
- connection readiness
- output readiness

Decision states:
- full_ready
- full_ready_with_warning
- manual_required
- blocked
