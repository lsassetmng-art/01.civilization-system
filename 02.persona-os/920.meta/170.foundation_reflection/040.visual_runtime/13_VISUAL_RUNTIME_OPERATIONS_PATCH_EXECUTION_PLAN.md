# 0. EXECUTION GOAL
# ============================================================

goal:
Apply VisualRuntime non-authoritative execution rules
to PersonaOS operations-layer documents so that
diagnostics, performance traces, render logs,
support traces, and operator-visible issues
do not overclaim canonical truth or verdict authority.

primary_outcome:
Operations-layer wording must stop collapsing:
- diagnostics into canonical result
- render logs into canonical visual truth
- performance success into canonical completion
- runtime error logs into security verdict
- operator-visible issues into authority records

- retry/support trace wording

# ============================================================
# 2. WHAT MUST BE ADDED
# ============================================================

must_add:
- diagnostics are evidence, not authority
- performance traces are support data
- render logs are not canonical truth
- operator-visible issues are not authority records
- support status is not canonical result
- observability wording must not overclaim commit/truth

must_add_fixed_rule:
VisualRuntime-related operations artifacts are support evidence
for observability and recovery only.

# ============================================================
# 3. WHAT MUST BE REMOVED OR TIGHTENED
# ============================================================

must_remove_or_tighten:
- wording that implies diagnostics equal canonical result
- wording that implies render failure equals canonical truth failure
- wording that implies performance success equals canonical completion
- wording that implies runtime logs equal security verdict
- wording that implies operator note equals governance/security/mutation record
- wording that implies support status equals Persona state authority

# ============================================================
# 4. PATCH METHOD
# ============================================================

patch_method:
1. inspect visual/runtime-related 070.operations documents
2. locate diagnostics wording, trace wording, log wording,
   dashboard wording, and support-status wording
3. insert evidence-vs-authority separation block
4. insert support-status non-authoritative block
5. replace overclaim wording with support-scoped wording
6. re-check consistency with first-pass runtime/architecture corrections
   and second-pass interface/security wording

# ============================================================
# 5. PREFERRED WORDING SHIFTS
# ============================================================

replace_like_this:

avoid:
- canonical failed
- truth invalid
- unauthorized output
- approval denied
- committed change failed
- state reverted
- authority record
- confirmed failure

prefer_when_support_only:
- diagnostics recorded
- render issue observed
- output unavailable
- fallback used
- performance degraded
- operator attention required
- runtime recovery pending
- support evidence recorded
- execution-side anomaly observed
