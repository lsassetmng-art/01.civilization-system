## 1. Purpose

Defines the boundary for handling rare exception cases
without breaking canonical governance.

## 2. Allowed Exception Scope

Exception handling may be used only for:

- temporary uncertainty during classification
- temporary uncertainty during traceability recovery
- rare documentation mismatch that needs clarification before routing

## 3. Not Allowed Under Exception Handling

Exception handling must not be used for:

- skipping canonical entry order
- bypassing governed change
- treating preserved candidate as approved entry
- overriding frozen baseline
- approving execution resume or transition implicitly

## 4. Exception Routing Rule

When an exception case appears:

- pause
- classify whether it is semantic or non-semantic
- route to recovery, maintenance, or governed change
- do not invent a new shortcut path

## 5. Hard Rule

Exception handling is a pause-and-route mechanism,
not an alternative governance path.

## 6. Acceptance Checklist

- allowed exception scope fixed
- not-allowed-under-exception fixed
- exception routing rule fixed
- no-alternative-governance-path rule fixed
